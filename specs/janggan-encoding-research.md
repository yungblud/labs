# 장단 인코딩 연구 토이 — "적은 데이터 × 도메인 인코딩" 연구 체험

> 출발점: arXiv 2408.01096 *Six Dragons Fly Again* (정간보 위치 인코딩 + 표준 트랜스포머로 궁중음악 복원).
> **목적은 결과물이 아니라 "이런 도메인의 연구 과정 자체를 손으로 한 바퀴 도는 것".**
> 후보 비교·확정 근거: [`music-ai-encoding-toys.md`](./music-ai-encoding-toys.md) (C안 확정).

---

## 목표

논문 저자들이 겪은 연구 루프를 **축소판으로 직접 재현**한다. 곡을 잘 뽑는 게 아니라, **"인코딩 설계가 적은 데이터에서 결과를 가른다"** 를 내 손으로 관찰·해석하는 게 성공 기준.

```
① 데이터 없는 도메인을 고른다 (제약이 곧 연구다)
② 표기법을 관찰해 인코딩을 내가 설계한다        ← 연구의 창의 지점
③ 작은 모델을 from scratch 로 학습한다
④ 인코딩을 바꿔가며 결과를 비교·측정한다         ← 연구의 실험 지점
⑤ "왜 이게 더 낫나" 를 해석해 노트로 남긴다      ← 연구의 결론 지점
```

**성공 판정:** ⑤의 노트가 나오면 성공. "생성곡이 얼마나 좋냐"는 부차적.

---

## 왜 C(장단)인가 (요지)

- **데이터 제약을 진짜로 재현** — ABC 민요는 데이터가 너무 많고 큰 LLM에 바로 먹혀서 인코딩 효과가 안 보인다. 장단은 **직접 소수만 손코딩** → 논문의 "limited data" 조건 재현.
- **인코딩 설계를 내가 발명** — 정간보식 "칸 안 위치 = 리듬" 을 리듬에만 떼어내 가장 순수하게 체험.
- **인코딩 차이가 극적으로 드러남** — 데이터가 적어야 "표현이 승부"라는 논문 메시지가 눈에 보인다.

---

## 스택 (확정)

- **Python.** from-scratch 소형 트랜스포머 학습은 Python 레퍼런스가 압도적 → 마찰을 인코딩 설계(②)에 몰아준다.
- 모델: **nanoGPT 급 초소형 GPT**(직접 최소 구현 or nanoGPT 축소). 데이터가 작아 CPU/맥에서 수 분 내 학습 목표.
- 데이터·인코딩 로직: 순수 Python(외부 의존 최소). 재생 검증 필요 시 `mido`/`pretty_midi` 로 MIDI 렌더(선택).

---

## 도메인 설계 — 장단을 어떻게 데이터로 만드나

정간보의 "한 칸(정간) = 한 박, 칸 안 위치 = 리듬 세분" 을 리듬 타악으로 축소한다.

- **대상 장단:** 굿거리 / 자진모리 / 세마치 등 **3~5종**. 각 장단의 기본형 + 변주 몇 개.
- **표기 단위:** 한 장단을 `박(칸)`들의 시퀀스로. 각 칸 안에서 타점의 **위치(0~N 등분)** 와 **악기(구음: 덩·기덕·쿵·따 등)** 를 적는다.
- **데이터 규모:** 손으로 **20~50 시퀀스** 제작(변주 포함). "데이터가 없어서 내가 만든다" = 연구 1단계 체험 그 자체.
- 원자료가 없으므로 **정답 검증 불가**를 그대로 받아들인다(논문과 동일 조건). 목표는 "장단다움" 재현이지 복원 정확도가 아니다.

---

## 인코딩 2종 (실험의 핵심 변수)

같은 장단 데이터를 **두 방식으로 토큰화**해서, 모델·데이터·학습을 동일하게 두고 **인코딩만 바꿔** 비교한다.

### v1 — naive(길이-기반 / flat)
- 타점을 절대 시간·길이로 나열. 서양식 duration 토큰에 대응.
- vocabulary 가 커지고, 칸 경계·박 구조 정보가 약함.

### v2 — 정간보식(위치-기반 / positional)  ← 논문 아이디어
- `|` = 칸(박) 경계, 칸 안 타점을 **위치 index(0~N)** + 구음으로.
- duration 을 위치로 흡수 → **vocabulary 축소**, 박 구조가 토큰에 내장.
- 논문의 JG-like 인코딩을 리듬 도메인으로 옮긴 것.

> 실험 가설(논문 재현): **적은 데이터에서 v2 가 v1 보다 "장단답게" 생성한다.**

---

## 실험 설계 (④)

- **고정:** 데이터셋 원본, 모델 크기, 학습 스텝, 시드. **변수는 인코딩(v1/v2) 하나만.**
- **측정(가벼운 정량 + 정성 병행):**
  - vocabulary 크기 / 토큰당 정보 밀도
  - 학습 loss 곡선(같은 데이터에서 어느 쪽이 빨리·낮게 수렴)
  - 생성 시퀀스의 **구조 위반율**(박 길이 안 맞음, 칸 경계 깨짐 등 규칙 기반 자동 체크)
  - 정성: 생성 장단 몇 개를 MIDI/텍스트로 보고 "장단다운가" 사람 판정
- **결론:** 차이의 원인을 인코딩 특성으로 해석.

## 체크리스트 (스텝별 — 각 스텝 후 다음 선언)

### 1단계 · 환경·뼈대 ✅

- [x] 코드 디렉터리 확정 — `janggan/` (스펙 `specs/` ↔ 코드 `janggan/`), 계획 구조는 `janggan/README.md`

### 2단계 · 데이터 제작 (연구 ①) ✅
- [x] 장단 4종 선정(굿거리·자진모리·세마치·**엇모리** 3+2+3+2 비대칭) + 표기 규칙 `data/NOTATION.md` (구음 5종·박×위치 그리드·shorthand)
- [x] 손저작 34 시퀀스 → `data/patterns.json` (재현 빌더 `data/build_dataset.py`, 무결성 검증 0위반)

### 3단계 · 인코딩 2종 (연구 ②) ✅
- [x] `v1_encode()` (flat/duration) + `v2_encode()` (positional/정간보식) — `janggan/encode.py`
- [x] 디코더 + 왕복 검증 — 34패턴 × 2인코딩 **실패 0** (v2 자기완결, v1 은 박길이 외부 주입 필요)

> **중간 관찰 (가설 수정):** 논문의 "위치 인코딩 → 어휘 축소"는 **이 데이터에선 재현 안 됨**
> (v1=8 < v2=10). 장단은 타점이 촘촘해 지속 길이가 d1~d3로 좁아 길이 토큰이 안 늘어난다.
> → 이 토이에서 v2 의 이점은 **어휘 크기가 아니라 "박 구조가 토큰에 명시됨"**(칸 경계 `|N`).
> **수정 가설:** v1 은 박 구조를 토큰에 못 담아, 생성 시 **박 길이 위반**이 더 잦을 것
> (특히 비대칭 엇모리). → 5단계 측정 초점을 어휘→**구조 위반율**로 이동.

### 4단계 · 학습 (연구 ③) ✅
- [x] 초소형 GPT — `janggan/model.py` (TinyGPT: 2층·d64·4head) + `dataset.py`(v1/v2 공용 텐서화)
- [x] v1/v2 동일 조건 학습 — `janggan/train.py` (seed 1337·2000step·동일 hp, 변수=인코딩만). `runs/*.pt`·`*.loss.json` 저장(gitignore)

> **관찰:** final loss v1=0.198 / v2=0.164. v2가 낮음 — 칸 경계·위치 명시로 다음 토큰이 더
> 예측 가능(모델링이 쉬움). 단 데이터가 작아 둘 다 사실상 암기 + 위치 불확실성 floor.
> 판정은 5단계 생성 실험(구조 위반율)에서.
