## 0. 원 논문에서 뽑은 재사용 레시피

복원이라는 겉모습이 아니라, **재현 가능한 핵심 레시피**만 추린다.

```
① 데이터가 극도로 적은 도메인을 고른다
② 그 도메인 고유의 "표기법 / 구조" 가 존재한다
③ 그 구조를 살린 인코딩(토큰화)을 직접 설계한다   ← 논문의 진짜 오리지널리티
④ 표준 트랜스포머(off-the-shelf)에 먹인다          ← 모델은 특별할 것 없음
⑤ 한 줄(입력) → 여러 파트(출력) 형태의 seq2seq
```

- 논문 성패는 **③ 인코딩 설계**에서 갈렸다. (음길이를 숫자가 아니라 정간 안 "위치 0~15"로 표현 → vocabulary 16개로 축소 → 적은 데이터로 학습 성공)
- **토이는 ①②③만 제대로 해도 논문 정신을 따라간 것.** 모델은 갖다 쓴다.
- OMR(이미지→데이터)은 논문 특유의 노가다 파트. **토이에선 이미 데이터셋이 있는 도메인을 골라 통째로 스킵**하는 게 정석.

### 무엇을 배우느냐 (조각별)

| 조각 | 원 논문 | 토이에서의 처리 |
| --- | --- | --- |
| OMR (이미지→구조 데이터) | CNN+트랜스포머 직접 학습, 합성 데이터 | **생략** — 공개 데이터셋 있는 도메인 선택 |
| 인코딩 설계 (③) | JG-like 위치 인코딩 | **여기 집중** — 표기법→토큰 변환 함수 하나 |
| 모델 (④⑤) | BERT류 + encoder-decoder | HuggingFace / nanoGPT / transformers.js |

## 1. 후보 리스트

각 후보: **무엇 / 왜 흥미로운가 / 데이터 / 난이도 / 논문과의 대응 / 배우는 것**.

### A. ABC notation 민요 생성 — "인코딩이 결과를 바꾼다"를 최단으로 체감  ⭐추천 시작점

- **무엇:** ABC notation(음악을 텍스트로 적는 표기법)으로 된 아이리시/스코틀랜드/한국 민요를 작은 언어모델에 학습시켜 새 곡 생성.
- **왜 흥미로운가:** ABC 자체가 이미 "위치·길이를 텍스트로" 표현하는 표기법 → **논문의 JG-encoding 발상을 그대로 체험**. 같은 곡을 다른 방식으로 토큰화해보고 결과가 갈리는 걸 직접 눈으로 봄.
- **데이터:** thesession.org (아이리시 민요 수만 곡, 공개).
- **난이도:** ★☆☆ (하루). 텍스트라 OMR·오디오 처리 전부 불필요.
- **논문 대응:** ③ 인코딩을 몸으로 배우는 최단 경로. ⑤(1줄→N파트)는 없지만 ③이 핵심이라 OK.
- **배우는 것:** 토큰화 방식이 생성 품질을 어떻게 바꾸는지. LLM 파인튜닝 미니 사이클.

### B. 코드 진행 → 편곡 (1줄 → N파트) — 논문 구조의 정확한 축소판

- **무엇:** `C–Am–F–G` 같은 코드 진행 한 줄 입력 → 베이스/드럼/패드 등 여러 파트 MIDI 자동 생성.
- **왜 흥미로운가:** **논문의 ⑤(한 줄 멜로디→6악기) 구조를 그대로 축소**. seq2seq "번역" 모델링을 직접 해봄.
- **데이터:** Lakh MIDI Dataset(공개, 대량) 또는 직접 만든 소수 예시.
- **난이도:** ★★☆ (주말). MIDI 파싱 + seq2seq. Python(pretty_midi) 필요.
- **논문 대응:** ③(코드/파트를 어떻게 토큰화) + ⑤(seq2seq) 둘 다 체험. 가장 논문에 가까움.
- **배우는 것:** 멀티파트 정렬을 인코딩에 어떻게 담는가(논문의 heterophony 문제 축소판).

### C. 한국 전통 리듬(장단) 시퀀스 생성 — "위치 인코딩"의 직계 후손

- **무엇:** 자진모리/휘모리 같은 장단(리듬 패턴)을 위치 기반 토큰으로 인코딩해 새 장단 생성/변주.
- **왜 흥미로운가:** 정간보의 "칸 안 위치=리듬" 아이디어를 **리듬만 떼어내 가장 순수하게** 재현. 음높이 복잡도를 빼서 ③에만 집중.
- **데이터:** 소수 장단 패턴을 직접 손코딩(데이터 극소 상황 = 논문 조건 재현).
- **난이도:** ★☆☆~★★☆. 데이터가 작아 인코딩 실험대로 딱.
- **논문 대응:** ③ 위치 인코딩의 본질을 가장 직접 계승. "적은 데이터" 조건도 진짜로 재현.
- **배우는 것:** 데이터가 거의 없을 때 인코딩·합성 데이터가 왜 승부처인지.

### D. COLDSURF 세트리스트 / 라인업 흐름 생성 — 내 도메인에 붙이기

- **무엇:** 공연 라인업·장르 한 줄 → "오프닝은 이 톤 / 미들 빌드업 / 클로징" 식 세트 흐름 or 추천 시퀀스 생성.
- **왜 흥미로운가:** 논문 레시피를 **음악이 아닌 내 실제 도메인 데이터(COLDSURF)** 에 이식. 토이가 프로덕트 실험으로 연결될 여지.
- **데이터:** COLDSURF 이벤트/라인업 데이터(내 것). 적음 → 논문 조건과 동일.
- **난이도:** ★★☆. 도메인 인코딩 설계가 관건(장르·순서·분위기를 토큰으로).
- **논문 대응:** ①(적은 데이터) ②③(도메인 구조→인코딩) 정통 계승. 소재만 비음악.
- **배우는 것:** "표기법이 없는 도메인에 표기법을 내가 발명"하는 경험. artist-engineer 결.

### E. 비음악 구조 채우기 (보너스) — 레시피가 음악 밖에서도 도는지

- **무엇:** 정규 구조가 있는 아무거나 → 나머지 채우기. 예: 이모지 시퀀스↔리듬, 코드 스니펫 한 줄→보일러플레이트.
- **왜 흥미로운가:** 논문 핵심(도메인 구조→토큰)이 음악과 무관하게 일반화됨을 확인.
- **난이도:** ★☆☆~★★☆ (소재 따라).
- **논문 대응:** ②③의 일반화 버전. ⑤도 소재에 따라 가능.
- **배우는 것:** "인코딩 설계 사고법"이 도메인 불문 전이되는지.
