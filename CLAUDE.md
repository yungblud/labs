# labs — 작업 지침

논문·연구를 **직접 손으로 재현하며 체험**하는 개인 실험 레포. 목적은 완성도 높은 프로덕트가 아니라 **연구 과정 자체를 한 바퀴 도는 것**. 레포 개요·진행 실험은 [`README.md`](./README.md).

## 성격 — 이 레포에서 무엇이 "성공"인가

- **결과물(생성 품질) ≠ 성공.** 실험의 성공은 각 스펙이 정의한 **관찰·해석(리서치 노트)** 이 나오는 것.
- 과도한 일반화·추상화·인프라 금지. **실험 하나를 최소 코드로 끝까지 도는 것**이 우선.
- paul-rockstar 본체 규약(앱·배포·도메인)은 **여기 적용 안 됨.** 독립 레포로 가볍게 간다.

## 구조

```
labs/
├── specs/        # 각 실험의 목표·설계·체크리스트 (작업 전 먼저 Read)
└── <experiment>/ # 실험별 코드 (예: janggan/)
```

- 새 실험은 `specs/<name>.md` 스펙부터. 스펙 없이 코드부터 시작하지 않는다.
- 코드는 실험 이름 폴더로 격리. 실험끼리 코드 공유·공통화하려 애쓰지 않는다(실험은 서로 독립).

## 작업 흐름

```
1. 스펙 Read → 목표·현재 스텝 확인
2. 방향이 갈리면 옵션 제시 → 사용자 결정 (임의 진행 금지)
3. 스텝 단위 실행 — 각 스텝 후 검증하고 다음 스텝 선언
4. 스펙 체크리스트 [x] 갱신
5. 커밋은 사용자가 명시적으로 요청할 때만
```

## 스택·툴링

- **Python** 기본 (from-scratch 소형 모델 실험 레퍼런스가 풍부). 무거운 의존은 피하고 표준 라이브러리 + numpy/torch 선.
- 가상환경은 `.venv/` (gitignore 됨). 모델 체크포인트·러닝 아티팩트도 커밋하지 않는다(gitignore 참조).
- 데이터·중간표현은 사람이 읽을 수 있는 형식(JSON 등)으로.

## 결정은 사용자에게

- 방향 전환·접근 변경·"이 정도면 충분한지" 같은 판단은 **추천만 하고 실행은 사용자 승인 후.**
- 자율 실행은 명시적으로 위임받은 스텝 범위 안에서만.
- 자동 커밋 금지 — 요청 있을 때만.
